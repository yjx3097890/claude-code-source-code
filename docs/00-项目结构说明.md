> 本文档基于从 npm 包 `@anthropic-ai/claude-code` v2.1.88 提取的反编译 TypeScript 源码。原始包是一个约 12MB 的单文件 `cli.js`。

## 核心架构

整个应用是一个经典的 AI Agent 循环，外层包裹了生产级的权限、流式传输、并发、上下文压缩、子代理、持久化和 MCP 协议支持。

```
用户输入 → messages[] → Claude API → 响应
                                       │
                              stop_reason == "tool_use"?
                             /                          \
                           是                            否
                            │                             │
                      执行工具                         返回文本
                      追加 tool_result
                      循环回去 ──────────────────→ messages[]
```


每个工具实现 `Tool` 接口，包含完整生命周期：

- `validateInput()` → 输入校验

```typescript
  | 'local_bash'           // 本地 Shell 命令
  | 'local_agent'          // 本地子代理

  | 'in_process_teammate'  // 进程内队友
```

## 数据流：单次查询生命周期

```
 用户输入 (prompt / 斜杠命令)
     │
     ▼
 processUserInput()                ← 解析 /commands，构建 UserMessage
     │
     ▼
 fetchSystemPromptParts()          ← tools → prompt sections, CLAUDE.md 记忆
     │
     ▼
 recordTranscript()                ← 持久化用户消息到磁盘 (JSONL)
     │
     ▼
 ┌─→ normalizeMessagesForAPI()     ← 剥离 UI 字段，必要时压缩
 │   │
 │   ▼
 │   Claude API (流式)             ← POST /v1/messages (tools + system prompt)
 │   │
 │   ▼
 │   流式事件                      ← message_start → content_block_delta → message_stop
 │   │
 │   ├─ text block ──────────────→ yield 给消费者 (SDK / REPL)
 │   │
 │   └─ tool_use block?
 │       │
 │       ▼
 │   StreamingToolExecutor         ← 分区：并发安全 vs 串行
 │       │
 │       ▼
 │   canUseTool()                  ← 权限检查 (hooks + rules + UI prompt)
 │       │
 │       ├─ 拒绝 ────────────────→ 追加 tool_result(error)，继续循环
 │       │
 │       └─ 允许
 │           │
 │           ▼
 │       tool.call()               ← 执行工具 (Bash, Read, Edit 等)
 │           │
 │           ▼
 │       追加 tool_result          ← push 到 messages[], recordTranscript()
 │           │
 └─────────┘                      ← 循环回 API 调用
     │
     ▼ (stop_reason != "tool_use")
 yield 结果消息                    ← 最终文本, usage, cost, session_id
```

## 构建系统

源码使用 Bun 的编译时内置函数，无法直接用 Node.js 编译：

- `feature('FLAG')` — Bun 编译时解析为 true/false，控制死代码消除 (DCE)
- `MACRO.X` — Bun 的 `--define` 编译时替换
- 108 个 feature-gated 模块在发布包中被 DCE 移除，不存在于 npm 包中

```
  Anthropic 内部 Monorepo              发布的 npm 包
  ──────────────────────               ─────────────────────
  feature('DAEMON') → true    ──构建──→  feature('DAEMON') → false
  ↓                                       ↓
  daemon/main.js  ← 包含      ──打包──→  daemon/main.js  ← 删除 (DCE)
```

项目提供了基于 esbuild 的尽力构建方案 (`scripts/build.mjs`)：

| 阶段 | 操作 |
|------|------|
| 1. 复制 | `src/` → `build-src/` (原始不动) |
| 2. 转换 | `feature('X')` → `false`，`MACRO.VERSION` → `'2.1.88'` |
| 3. 入口 | 创建注入 MACRO 全局变量的包装器 |
| 4. 打包 | esbuild 迭代打桩 + 打包，可达约 95% 覆盖率 |

## 关键设计模式

1. **`buildTool()` 工厂** — 统一的工具定义和生命周期管理，每个工具通过声明式定义创建
2. **`StreamingToolExecutor`** — 并发安全的工具分区执行：只读工具并行，写入工具串行
3. **`autoCompact()`** — 基于 token 预算的自动上下文压缩，防止超出上下文窗口
4. **GrowthBook feature flags** — 远程功能开关，支持 A/B 测试和渐进发布
5. **双分析管道** — 第一方 (Anthropic) + Datadog，两路遥测数据上报
6. **MCP 协议** — Model Context Protocol，标准化的外部工具/资源集成
7. **React/Ink 终端 UI** — 基于 React 的终端渲染框架，支持虚拟滚动、diff 显示、搜索高亮
8. **权限分层** — hooks → rules → interactive prompt → tool-specific check，四层权限防护
9. **任务 ID 前缀** — 类型化的任务标识 (b/a/r/t/d)，36^8 ≈ 2.8 万亿组合防碰撞
10. **会话持久化** — JSONL 格式的 transcript 记录，支持 resume 和跨会话搜索
