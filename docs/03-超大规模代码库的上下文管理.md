# 超大规模代码库的上下文管理深度分析

> 基于 Claude Code v2.1.88 反编译源码，解析其如何在有限的上下文窗口中管理超大规模代码库。

---

## 一、问题的本质

Claude Code 面临的核心矛盾：

- 代码库可能有数百万行代码
- 模型上下文窗口最大 200K tokens（实验性 1M）
- 每轮对话都会累积工具调用结果，快速消耗上下文
- 用户期望长时间连续工作，不想频繁重启会话

Claude Code 的解决方案是一个多层次的上下文管理体系，从预防到恢复，层层递进。

## 三、多层压缩策略

Claude Code 实现了五种不同的上下文压缩机制，按执行顺序排列：

### 3.1 Tool Result Budget（工具结果预算）

在所有压缩之前，先对单条工具结果的大小进行限制：

```typescript
messagesForQuery = await applyToolResultBudget(
  messagesForQuery,
  toolUseContext.contentReplacementState,
  persistReplacements ? records => void recordContentReplacement(...) : undefined,
  // 排除没有大小限制的工具
  new Set(tools.filter(t => !Number.isFinite(t.maxResultSizeChars)).map(t => t.name)),
)
```

这是最细粒度的控制——防止单个巨大的工具结果（如读取大文件）占满上下文。

### 3.2 History Snip（历史裁剪）

```typescript
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
}
```

Snip 是一种轻量级的压缩——直接移除旧消息，不生成摘要。它比完整压缩快得多，但会丢失信息。

### 3.3 MicroCompact（微压缩）

```typescript
const microcompactResult = await deps.microcompact(
  messagesForQuery, toolUseContext, querySource
)
```

MicroCompact 有两种模式：

**标准模式**：替换旧的工具结果为简短摘要

```typescript
function collectCompactableToolIds(messages: Message[]): string[] {
  // 收集可以被压缩的工具结果 ID
  // 保留最近的 N 个结果不压缩
}
```

**缓存友好模式** (Cached MicroCompact)：利用 API 的 cache_deleted_input_tokens 特性

```typescript
// 不修改消息内容，而是通过 API 的缓存编辑机制删除旧的工具结果
// 这样不会破坏 prompt cache
const pendingCacheEdits = microcompactResult.compactionInfo?.pendingCacheEdits
```

### 3.4 Context Collapse（上下文折叠）

```typescript
if (feature('CONTEXT_COLLAPSE') && contextCollapse) {
  const collapseResult = await contextCollapse.applyCollapsesIfNeeded(
    messagesForQuery, toolUseContext, querySource
  )
  messagesForQuery = collapseResult.messages
}
```

Context Collapse 是一种"读时投影"机制——它不修改 REPL 的完整历史，而是在每次 API 调用前生成一个折叠视图。折叠的摘要存储在独立的 collapse store 中，跨轮次持久化。

折叠在自动压缩之前运行，这样如果折叠就能把 token 数降到阈值以下，就不需要触发完整压缩。

### 3.5 AutoCompact（自动压缩）

这是最重量级的压缩机制，会调用 Claude API 生成对话摘要：

```typescript
async function autoCompactIfNeeded(messages, toolUseContext, ...) {
  // 熔断器：连续失败 3 次后停止重试
  if (tracking?.consecutiveFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES) {
    return { wasCompacted: false }
  }

  if (!await shouldAutoCompact(messages, model, querySource, snipTokensFreed)) {
    return { wasCompacted: false }
  }

  // 优先尝试 Session Memory Compaction（更轻量）
  const sessionMemoryResult = await trySessionMemoryCompaction(
    messages, agentId, autoCompactThreshold
  )
  if (sessionMemoryResult) return { wasCompacted: true, compactionResult: sessionMemoryResult }

  // 回退到完整压缩
  const compactionResult = await compactConversation(messages, toolUseContext, ...)
  return { wasCompacted: true, compactionResult }
}
```

## 四、压缩提示词的设计

### 4.1 压缩提示词结构

压缩提示词要求模型生成一个结构化的摘要，包含 9 个部分：

```
1. Primary Request and Intent — 用户的明确请求和意图
2. Key Technical Concepts — 关键技术概念
3. Files and Code Sections — 文件和代码段（含完整代码片段）
4. Errors and fixes — 错误和修复
5. Problem Solving — 问题解决过程
6. All user messages — 所有用户消息（非工具结果）
7. Pending Tasks — 待完成任务
8. Current Work — 当前工作（最近的消息）
9. Optional Next Step — 可选的下一步
```

### 4.2 分析-摘要模式

```
Before providing your final summary, wrap your analysis in <analysis> tags...

<analysis>
[思考过程 — 确保覆盖所有要点]
</analysis>

<summary>
[最终摘要]
</summary>
```

`<analysis>` 块是一个"草稿本"——它提高了摘要质量，但在 `formatCompactSummary()` 中被剥离，不进入最终上下文：

```typescript
function formatCompactSummary(summary: string): string {
  // 剥离 analysis 部分 — 它是提高质量的草稿，没有信息价值
  formattedSummary = formattedSummary.replace(
    /<analysis>[\s\S]*?<\/analysis>/, ''
  )
  // 提取 summary 部分
  const summaryMatch = formattedSummary.match(/<summary>([\s\S]*?)<\/summary>/)
  // ...
}
```

### 4.3 禁止工具调用

压缩提示词的开头和结尾都强调不要调用工具：

```typescript
const NO_TOOLS_PREAMBLE = `CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
- Do NOT use Read, Bash, Grep, Glob, Edit, Write, or ANY other tool.
- You already have all the context you need in the conversation above.
- Tool calls will be REJECTED and will waste your only turn — you will fail the task.`

const NO_TOOLS_TRAILER = `REMINDER: Do NOT call any tools. Respond with plain text only —
an <analysis> block followed by a <summary> block.
Tool calls will be rejected and you will fail the task.`
```

源码注释解释了原因：在 Sonnet 4.6+ 上，模型有时会尝试调用工具，导致 `maxTurns: 1` 下没有文本输出，回退到流式 fallback（2.79% 的失败率）。

### 4.4 部分压缩

除了完整压缩，还支持部分压缩（只压缩旧消息，保留最近的消息）：

```typescript
// 'from' 方向：压缩最近的消息，保留旧的上下文
const PARTIAL_COMPACT_PROMPT = `Your task is to create a detailed summary
of the RECENT portion of the conversation — the messages that follow
earlier retained context.`

// 'up_to' 方向：压缩旧消息，保留最近的消息
const PARTIAL_COMPACT_UP_TO_PROMPT = `Your task is to create a detailed
summary of this conversation. This summary will be placed at the start
of a continuing session; newer messages that build on this context will
follow after your summary.`
```

## 五、Session Memory Compaction

这是一种比完整压缩更轻量的替代方案：

```typescript
// src/services/compact/sessionMemoryCompact.ts
async function trySessionMemoryCompaction(
  messages, agentId, autoCompactThreshold
) {
  if (!shouldUseSessionMemoryCompaction()) return null

  // 计算需要保留的消息索引
  const keepIndex = calculateMessagesToKeepIndex(messages, ...)

  // 调整索引以保持 API 不变量（user/assistant 交替）
  const adjustedIndex = adjustIndexToPreserveAPIInvariants(messages, keepIndex)

  // 直接裁剪旧消息，不生成摘要
  return createCompactionResultFromSessionMemory(messages, adjustedIndex)
}
```

Session Memory Compaction 的优势：
- 不需要额外的 API 调用（不生成摘要）
- 速度快（直接裁剪）
- 依赖 CLAUDE.md 记忆系统保留关键上下文

## 八、反应式恢复

### 8.1 Reactive Compact

当 API 返回 prompt-too-long 错误时的紧急压缩：

```typescript
// 在流式循环中，prompt-too-long 错误被"扣留"而非立即显示
let withheld = false
if (reactiveCompact?.isWithheldPromptTooLong(message)) {
  withheld = true  // 不 yield 给消费者
}

// 流式结束后，尝试恢复
if (isWithheld413 && reactiveCompact) {
  const compacted = await reactiveCompact.tryReactiveCompact({
    hasAttempted: hasAttemptedReactiveCompact,
    messages: messagesForQuery,
    // ...
  })
  if (compacted) {
    // 用压缩后的消息重试
    state = { ...state, messages: buildPostCompactMessages(compacted) }
    continue
  }
  // 恢复失败 → 显示错误
  yield lastMessage
  return { reason: 'prompt_too_long' }
}
```

## 九、关键设计模式总结

| 层次 | 机制 | 时机 | 成本 |
|------|------|------|------|
| L0 | Tool Result Budget | 每次工具执行后 | 零（本地截断） |
| L1 | History Snip | 每轮 API 调用前 | 零（直接裁剪） |
| L2 | MicroCompact | 每轮 API 调用前 | 零/低（本地替换或缓存编辑） |
| L3 | Context Collapse | 每轮 API 调用前 | 零（读时投影） |
| L4 | Session Memory Compact | 达到阈值时 | 零（直接裁剪 + 记忆系统） |
| L5 | AutoCompact | 达到阈值时 | 高（额外 API 调用生成摘要） |
| L6 | Reactive Compact | API 返回 413 时 | 高（紧急 API 调用） |

设计原则：
1. **渐进式**：从零成本到高成本，逐级升级
2. **缓存友好**：尽量不破坏 prompt cache
3. **熔断保护**：连续失败 3 次后停止重试
4. **信息保留**：压缩摘要包含 9 个结构化部分，最大化信息保留
5. **无缝续接**：压缩后指示模型"像中断从未发生一样继续"
